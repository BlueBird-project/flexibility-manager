# Description of the project

The BlueBird Project implements demand response for different type of local energy assets :
* Water Work (Water Tower) 
* Building HVAC systems
* Sport center and swimming pool 
* Small scale district heating 

We call the different models a pilot and they are spread accross the European Union and partners of the BlueBird Project. 

# Modelling and Control Description

* The control is based on a model of the assets. We call such a model the Digital Twin. 
* The control can be any form of sequential decision making. We will most often use economic MPC. But it could range Reinforcement Learning, Approximate Dynamic Programming, Stochastic Programming or more. 
* The controller receives a model of the system as form of state-dynamic equations (preferably linear with boolean variables). It receives the matrices describing the evolution $\dot x = A x(t) + B u(t) + E d(t)$
* The controller might directly read measurement from the database or use the measurement for state estimation (Kalman Filter not developped yet).
* The controller to work uses forecasts of the disturbances and electricity market signal. 
* The forecasts of the disturbance and the service handleing electricity market signals (called the trading manager) are two seperated services. 
* Depending on the pilot and context, the trading manager could use forecasts of the electricity price. Or directly the result of the Day-ahead clearing price. 
* This sums up to 4 seperated services running independently and sharing information. 1. Digital Twin. 2. Flexibility Controller. 3. Disturbances Forcast. 4. Trading Manager. All 4 services should be built as seperate containers. 

# Implementation Description

* All but the Trading Manager service are implemented in this folder. The trading manager is implemented in a seperate git repository and runs as a service by building the containers. An example of getting prices from the trading manager can be found in @FlexOPTi/scripts/tm/query_prices.jl
* The prices from the trading manager are fetched once every 15 minutes. For day-ahead market they will not update, but this is to check if the market give corrections. Those prices are stored in a table in the database. 
* The Disturbances Forecast Service will fill the database with forecasts, it will periodically update the forecasts. For the moment this service is not available,  so dummy forecasts can be considered as it is done for the room temperature in ewh pilot (@FlexOPTi/scripts/ewh/mpc_base.jl). But it should already opperate as final. 
* The digital Twin has two actions. It responds to the flexibility controller when a request for updating its dynamics when asked. Once the dynamics is updated it responds back to the flexibilty manager offering it's dynamic. The way this dynamic information reads is given in @FlexOPTi/data/EWH/inputs/cold_room_model_continuous.json. This file should be writtin in a shared volume between the digital twin and the flexibility controller. There should be in fact one shared volume accross all the services. The flexbility controller should at each MPC itereation read the lattest file in the shared folder.    
* The flexibility controller API is described in @FlexOPTi/src/core/fm_optimize.jl and can be called with a bunch of different keyword arguments. docker compose should consider a .env and settings.json that contains those arguments for the opperator to simply change the setting.json file seamlessly. 
* The flexibility controller API uses julia multiple dispatch to distinguish between pilots. A datatype in @FlexOPTi/src/core/types.jl enters as argument of the main functions. In the API entry point @FlexOPTi/src/core/fm_optimize.jl, this is handled as a keyword argument where one of the argument is the pilot given as a string. 
* The database is a simple SQLite database. 

* Control room : Eventually I would like to have a virtual control room (docker itself or bash script or julia or python script) that can do different things :
- Run the system (so run the MPC every 15 minutes, all the services)
- Compute KPIs (Key performance metrics)
- Fit flexibility function 
- Run a single step
- Plot some results 
- etc.
* Such a control room will allow to do reasearch at the same time as having a system opperating. 
* No need to build this in details for now, but just set the infrastructure.
