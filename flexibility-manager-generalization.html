Architecture

Proposed config-driven flow

Design principle

Everything currently encoded as pilot-specific logic becomes declarative configuration. The generic MPC kernel reads config, builds JuMP dynamically, solves, and writes semantic results.

Inputs
Pilot config, digital twin, sensors DB, forecasts, trading manager.
Engine
Read config → parse DT → load sensors → load forecasts → build JuMP → solve → write setpoints.
Output schema
(timestamp, zone_id, control_name, value)
pilot_config.json ├─ zones ├─ controls ├─ disturbances ├─ dynamics ├─ power_topology ├─ objective ├─ market ├─ mpc ├─ retraining └─ database Generic MPC Engine 1. Reads pilot_config 2. Parses DT 3. Reads sensors 4. Reads forecasts 5. Builds JuMP 6. Solves 7. Writes generic setpoints
Pilot config

`pilot_config.json` becomes the system contract

Zones
State variables, limits, and setpoint ranges per zone

Examples: dairy, freezer, zone_10; T_air, T_product, T_room.

Controls & disturbances
Continuous / binary controls and global / per-zone disturbances

Examples: p1, p2, p3, sp_hvac; T_ambient, occupancy, solar_irradiance.

Dynamics & topology
Model type, vector layouts, compressor groups, objective, market, DB

The engine does not need to know whether the pilot is EWH or Montcada.

EWH example
5 cold storage rooms, shared compressor groups, state-space model, energy-cost objective.
Montcada example
38 HVAC zones, ARX dynamics, continuous HVAC setpoint, mode switching in topology.
Core engine

FlexOPTi refactor: generic structs, parser, constraints

Types

Replace concrete pilot structs with GenericPilot + PilotConfig and typed config objects such as ZoneConfig and ControlConfig.

Digital twin parser

Dispatch by dynamics.type, e.g. :state_space or :arx, while preserving pilot-specific DT JSON structures.

Dimensional validation

Validate matrix sizes against config layouts: A(nx,nx), B(nx,nu), E(nx,nd).

Constraint loading

Build limits directly from config for zone bounds, setpoints, and control min/max values.

MPC builder

Dynamic JuMP model assembly

Variables
States + setpoints + typed controls

Continuous controls get bounds; binary controls respect MILP horizon and relax afterwards if needed.

Dynamics
`X = Ξ·U + M·ξ₁ + Ψ·Δ`

Control and disturbance vectors are built from layouts, not file-specific assumptions.

Comfort
Bounds per observed state

For every zone and every constrained state name in config.

Objective
Composable cost terms

Grid buy / sell terms resolved from a price source such as the trading manager.

Critical point
This is the structural shift: the model is no longer authored once per pilot; it is assembled from declarative metadata and reusable topology blocks.

Generic / future pilots.
Implementation pattern
A Julia registry maps group.type to the corresponding builder function, so adding a topology means adding a new block, not rewriting the MPC kernel.
Production controller & DB

Generic operations layer: sensors, setpoints, and storage

Generic sensor read
Build SQL columns dynamically from the configured state layout and reconstruct x0 by zone/state mapping.
Generic setpoint write
Write first-step actions using (timestamp, zone_id, control_name, value) for each control mapping.
DB redesign
Move to EAV-style tables for measurements, disturbance_forecasts, and setpoints.
Why EAV
Same tables for any pilot, no schema migration for new variables, easy filtering by zone_id and var_name.
Trade-off
A fixed-column alternative is faster but less flexible; the recommendation is EAV for maximum pilot portability.
Forecast / dashboard / Docker

The outer services must become config-aware too

Forecast service
Read disturbances from config

Write global and per-zone fallback forecasts from disturbances and zones.

Control Room
Build views dynamically

Pilot name, zones, controls, and recent setpoints come from the active config.

Docker
Single compose file

Use PILOT_ID to switch configs and optional DT Dockerfile/retraining profile.

Operational effect
Switching from EWH to Montcada—or onboarding a new water works pilot—should be an environment/config decision rather than a branch of code.
Implementation plan

Four phases with increasing structural impact

Phase 1
Config-driven constraints
Create EWH config, load it in Julia, rewrite constraint loading, keep current MPC intact, validate with identical numerical results.
Phase 2
Generic DB + parameterized controller
Move to generic EAV schema, rewrite DB read/write, adapt dummy forecast and dashboard to config.
Phase 3
GenericPilot + dynamic MPC
Refactor types, DT parser, variable builder, generic constraints, objective, batch disturbances, and generic output packaging.
Phase 4
Generic Docker + onboarding
Unify compose, document new pilot onboarding, provide config template, and add JSON Schema validation.
Validation checklist

Definition of done for the generalized framework

Regression
EWH must produce exactly the same setpoints as before.
Zero-code onboarding
A new pilot must be definable using only pilot_config.json.
Generic persistence
DB schema must support any mix of zones and variables.
Dynamic UI and forecast
Dashboard and dummy forecast must reflect the active config automatically.
Deployment switch
Changing pilots must be possible with PILOT_ID.
Safety net
JSON Schema validation and unit tests must be parameterized with example configs.
